## What this project is

Lightweight Windows screenshot/burst-capture tool. The goal is faster cold start than Snipping Tool / ShareX. Three build paths exist; **native (C) is the fast path** — the others are kept for compatibility and to reduce JIT cost.

## Self-tests

Run before committing capture-path changes.

```powershell
.\build\ㅋㅊ.exe --self-test
.\build\ㅋㅊ.exe --self-test-dxgi
.\build\netfx\ㅋㅊ.exe --self-test
.\build\dotnet-r2r\ㅋㅊ.exe --self-test
```

Exit codes: `0` pass, `1` capture/save failed, `2` DXGI path did not run or second burst failed.
`--self-test-dxgi` runs display burst twice in one process to verify the D3D11 device cache.

## Hard rules

### Native (`native/wincap_native.c`)

- Built with `/NODEFAULTLIB` + custom `wWinMainCRTStartup`. **Do not add CRT-dependent headers** (`<stdio.h>`, `<string.h>`, `<stdlib.h>`, etc.). The hand-rolled `memset`/`memcpy` registered via `#pragma function` is required by the compiler's intrinsic lowering — do not delete.
- DLL loading: always `LoadLibraryExW(..., LOAD_LIBRARY_SEARCH_SYSTEM32)`, never plain `LoadLibraryW` (DLL planting hardening).
- Output format: BMP. Adding PNG would pull in `gdiplus` or a third-party encoder — that defeats the no-codec design. If you need compression, propose it as a separate discussion.
- Binary size baseline ~30 KB. Heavy compiler flags (`/Zi`, `/RTC*`, `/MD*`) will balloon this — verify after touching `build-native.ps1`.

### Burst hot path

- Use `AllocMem` (no zero) for buffers about to be fully overwritten. `AllocZero` is only for genuinely zero-initialized memory. Mixing these up costs an extra full-buffer scan per allocation.
- `BurstFrameStore` is one contiguous `block` keyed on `frameBytes` — frame size must stay constant for the duration of the burst.
- `BeginBurstPerfScope` enters `timeBeginPeriod(1)` + MMCSS `"Capture"` + `HIGH_PRIORITY_CLASS`. `EndBurstPerfScope` reverts. Don't add long sleeps to the burst loop without this scope, and don't omit the End call.
- `SleepUntilTick` uses `QueryPerformanceCounter`-based wait near the target tick. Don't replace with plain `Sleep(interval)` — Windows `Sleep` resolution defaults to ~15.6 ms without `timeBeginPeriod(1)`.

### DXGI display path

- Device/context/output1 are cached in `g_dxgiCache`, keyed by `RECT bounds`. `StoreDxgiDeviceCache` AddRefs all three; `ReleaseDxgiDeviceCache` releases. Audit refcounts when editing this path.
- `IDXGIOutputDuplication::AcquireNextFrame(0)` (zero timeout) is intentional — fixed-interval contract. A positive timeout will jitter the recording schedule.
- `DXGI_ERROR_WAIT_TIMEOUT` means "no new frame since last call" — keep the prior `lastFrame` buffer and continue. Don't treat it as an error.
- DXGI accelerates **only** `MODE_DISPLAY` burst. The `TryCaptureBurstFramesDxgiDisplay` function returns FALSE for other modes and the caller falls back to GDI. Don't remove the GDI fallback.
- The path is skipped for rotated displays (`outputDesc.Rotation != DXGI_MODE_ROTATION_IDENTITY`).

### `CAPTUREBLT` flag

- Default off. Toggled via the "Layered" UI button → `g_includeLayeredWindows` → `GetCaptureRop()`. Re-enabling by default reintroduces a DWM sync round-trip on every `BitBlt`.

### Freeform mask

- `ApplyFreeformMaskWithNodes` is a scanline polygon fill (sort + span). **Do not replace** with per-pixel `PointInsidePolygon` — that's the O(W·H·N) version that was removed for performance.
- The `yi != yj` guard before the slope calc prevents div-by-zero on horizontal polygon edges. Keep it.

### Managed builds (`src/`)

- `FastCaptureScope` no longer sets `GCLatencyMode.LowLatency`. Don't restore it — it caused LOH pressure issues in long bursts. If you need a managed no-GC region, propose `GC.TryStartNoGCRegion` with sizing logic instead.
- Managed builds save PNG, native saves BMP. This asymmetry is intentional (no codec dep in native).
- Clipboard copy publishes both `Bitmap` and DIB formats for broad paste compatibility — see `ClipboardUtil.cs`.

## Common task patterns

### Adding a capture-path optimization
1. Decide which build(s) to touch. Native is the fast path; changes there have the largest impact.
2. Run all four self-tests after the change. `--self-test-dxgi` exercises the D3D11 cache reuse path.
3. If the change affects timing (intervals, priorities, sleeps), test with a real burst — self-tests run a 1-frame burst and won't catch jitter regressions.

### Adding a UI control to the native build
- Define a unique `IDC_*` constant (current range 1001–1014).
- Create the HWND in `CreateMainControls`.
- Add layout in `LayoutMain`. The toolbar wraps when the window is narrow — verify wrapping behavior.
- Wire up `WM_COMMAND` handling in `MainProc`.

### Updating the README
README is bilingual (Korean + English) with parallel sections. When editing, update both languages. The section list is mirrored exactly — don't let them drift.

## What's intentionally not done

- **Windows Graphics Capture (WGC)**: not used. WinRT interop in `/NODEFAULTLIB` C is expensive (HSTRING, IInspectable, async handlers, ~500–800 LOC of glue). The real win would be window-mode correctness (occluded windows, hardware overlays) — not speed. If pursuing, do it in the R2R build via `Windows.SDK.Contracts` projection, not native.
- **DXGI for window/rectangle modes**: not implemented. Would require monitor-bounds detection, ROI crop, and cross-monitor split handling. The current GDI path is the deliberate trade-off.
- **Frame-skip / delta detection**: not implemented. Reasonable next step if burst storage becomes an issue (xxhash 4×4 downsample compare).
- **Tray-resident / pre-warm mode**: explicitly called out in README as the strongest cold-start option but not implemented. Architecturally larger than any capture-path optimization.
- **WM_DISPLAYCHANGE handler**: not implemented. The `bounds`-keyed DXGI cache naturally invalidates on resolution change, and `DuplicateOutput` failure self-heals. Adding the handler is nice-to-have, not required.
